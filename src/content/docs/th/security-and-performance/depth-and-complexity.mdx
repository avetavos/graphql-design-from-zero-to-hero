เราได้ตกลงเรียบร้อยแล้วว่า *ใคร* เป็นคนถามและ *เขาได้รับอนุญาตหรือไม่* ตอนนี้มาถึงคำถามข้อที่สาม: **query นี้แพงแค่ไหน?** คำขอหนึ่งอาจ authenticate ครบถ้วน, authorize เต็มที่ และยังทำให้ server ล้มลงคุกเข่าได้ — เพราะ GraphQL ให้ client เขียน *รูปทรง* ของงานเอง และ query string เล็ก ๆ สามารถอธิบายงานปริมาณมหาศาลได้

## query เล็ก ๆ กลายเป็น denial of service ได้อย่างไร

สองฟีเจอร์ของ graph ที่เชื่อมโยงกันทำให้การใช้ในทางที่ผิดเป็นเรื่องง่าย:

- **การซ้อนลึก** ถ้า `User` มี `friends: [User!]!` แล้ว `user → friends → friends → friends → …` ก็เป็น SDL ที่ถูกต้องได้ไม่รู้จบ แต่ละชั้นทวีคูณจำนวน row ที่ server ต้องดึง สิบชั้นของ fan-out ขนาดปานกลางคือ object หลายล้านชิ้นจาก query ไม่กี่บรรทัด
- **Alias** client อาจขอ field ที่แพงตัวเดิมหลายครั้งภายใต้ชื่อต่างกัน: `a: search(...) b: search(...) c: search(...)` query ยังคงสั้น แต่ server รันงานหนึ่งครั้งต่อหนึ่ง alias

```graphql
# Short to write, ruinous to execute: nesting plus aliasing.
query Abuse {
  user(id: "1") {
    friends {           # level 1
      friends {         # level 2
        friends {       # level 3 ... and on, and on
          a: posts { title }
          b: posts { title }   # same field, billed twice via aliases
        }
      }
    }
  }
}
```

คุณแก้สิ่งนี้ด้วย authentication หรือ authorization ไม่ได้ — ผู้โจมตีอาจเป็น user ที่ถูกต้องตามกฎหมายและล็อกอินแล้ว คุณแก้มันด้วยการวัด cost และปฏิเสธสิ่งที่แพงเกินไป *ก่อน execution*

## สามแนวป้องกัน เริ่มจากที่ถูกที่สุด

1. **Depth limiting** เดินผ่าน query ที่ parse แล้วและปฏิเสธมันหากการซ้อนเกินขีดสูงสุดที่กำหนด (เช่น 7) มันหยาบแต่ถูกและจับการโจมตีแบบซ้อนเรียกซ้ำได้ทันที นี่เป็นตัวควบคุมที่เพิ่มได้ง่ายที่สุดและเป็นตัวที่เราจะคำนวณสด ๆ ด้านล่าง
2. **Complexity (cost) analysis** กำหนด cost ให้แต่ละ field — ตัวเลขคงที่ หรือสูตรที่อิงกับ argument อย่าง `first: 100` สำหรับ list ที่ทำ pagination — รวม cost ทั้ง query, คูณผ่านการซ้อน, และปฏิเสธอะไรก็ตามที่เกินงบประมาณ วิธีนี้จัดการได้ทั้ง query ที่กว้าง *และ* ลึก และคำนึงถึง alias เพราะแต่ละ alias เป็น selection แยกต่างหากที่บวกเข้ากับยอดรวม
3. **Timeout** เป็นกองหลังสำหรับอะไรก็ตามที่หลุดผ่าน static analysis: จำกัด wall-clock execution (และ call ต่อ data source) เพื่อให้ query ที่กลายเป็นว่าแพงตอน runtime ถูกยกเลิกแทนที่จะปล่อยให้รันไปตลอด

ไลบรารีอย่าง `graphql-depth-limit` และ `graphql-query-complexity` implement สองตัวแรกเป็น validation rule ดังนั้น query จึงถูกปฏิเสธระหว่าง validation — ไม่มี resolver ตัวใดรันเลย กฎ cost อ่านได้แบบนี้:

```ts
import { createComplexityRule, simpleEstimator } from 'graphql-query-complexity';

const complexityRule = createComplexityRule({
  maximumComplexity: 1000,
  estimators: [
    // A list field costs (childComplexity * the requested page size).
    simpleEstimator({ defaultComplexity: 1 }),
  ],
  onComplete: (cost) => {
    if (cost > 1000) throw new Error('Query is too complex: ' + cost);
  },
});
// Pass complexityRule in the server's validationRules so it runs before execution.
```

## คำนวณ depth จาก query ที่ parse แล้ว แบบสด ๆ

Depth limiting เป็นสิ่งที่คุณ implement ได้ในไม่กี่บรรทัด และหัวใจของมันคือฟังก์ชัน `parse` ของ GraphQL — ตัวเดียวกับที่ server ใช้ มันเปลี่ยน query string ให้เป็น **AST** (abstract syntax tree) เดินผ่าน `selectionSet` ทั้งหลาย แล้วสายโซ่ของ selection ซ้อนที่ลึกที่สุดก็คือ depth ของ query demo ด้านล่าง parse สาม query แล้ววัดแต่ละตัว จากนั้นใช้ limit เท่ากับ 5 กด **Run**

export const depthCode = `(async () => {
  const { parse } = await import('https://esm.sh/graphql@16');

  // Depth = the deepest chain of nested selection sets in the operation.
  function selectionDepth(selectionSet, current) {
    let max = current;
    for (const sel of selectionSet.selections) {
      if (sel.selectionSet) {
        max = Math.max(max, selectionDepth(sel.selectionSet, current + 1));
      } else {
        max = Math.max(max, current + 1);
      }
    }
    return max;
  }

  function queryDepth(source) {
    const doc = parse(source);
    let max = 0;
    for (const def of doc.definitions) {
      if (def.kind === 'OperationDefinition') {
        max = Math.max(max, selectionDepth(def.selectionSet, 0));
      }
    }
    return max;
  }

  const MAX_DEPTH = 5;

  const queries = {
    shallow: \`{ user { name } }\`,
    medium: \`{ user { friends { posts { title } } } }\`,
    abusive: \`{
      user {
        friends { friends { friends { friends { friends { name } } } } }
      }
    }\`,
  };

  for (const [label, source] of Object.entries(queries)) {
    const depth = queryDepth(source);
    const verdict = depth > MAX_DEPTH ? 'REJECTED' : 'allowed';
    console.log(label + ': depth ' + depth + ' -> ' + verdict + ' (limit ' + MAX_DEPTH + ')');
  }
})();`;

<Callout title="วัด cost แล้วปฏิเสธก่อนที่คุณจะ execute">
query หนึ่งอาจผ่าน authentication และ authorization และยังเป็นอาวุธ denial-of-service ได้ เพราะ client ควบคุมรูปทรงของงาน จงป้องกันเป็นชั้น ๆ: **depth limiting** หยุดการซ้อนที่ควบคุมไม่ได้, **complexity analysis** จัดงบประมาณ cost รวม (นับ argument และ alias), และ **timeout** จับอะไรก็ตามที่กลายเป็นว่าแพงตอน runtime สองตัวแรกเป็น static — มันอ่าน query ที่ parse แล้วและปฏิเสธระหว่าง validation ดังนั้น resolver จึงไม่เคยรัน จงตั้ง maximum depth และ complexity budget บนทุก public GraphQL endpoint เสมอ; endpoint ที่ไม่มีสิ่งเหล่านี้ก็อยู่ห่างจาก outage เพียง query ฉลาด ๆ หนึ่งตัวเท่านั้น
</Callout>
