นี่คือจุดที่ทุกอย่างมาลงตัว คุณได้ออกแบบ schema, เขียน resolvers, เพิ่ม mutations, จัดการ pagination และ errors, ทำให้กราฟปลอดภัย และเรียนรู้วิธีใส่ type, test และ federate มันมาแล้ว บทเรียนนี้ประกอบแนวคิดเหล่านั้นเข้าเป็น TypeScript service เล็ก ๆ ที่เป็นหนึ่งเดียวด้วย **GraphQL Yoga** — GraphQL server แบบ batteries-included ที่รันได้ทุกที่ที่ Node รันได้ — และมอบโปรเจกต์ที่รันได้จริงให้คุณนำติดตัวไป

## สามชิ้นส่วนของ Yoga service

Yoga server สร้างจากสามชิ้นส่วนแบบเดียวกับที่ GraphQL server ทุกตัวมี เพียงแต่ตอนนี้ผูกเข้าด้วยกันอย่างชัดเจน:

- **typeDefs** — schema ในรูป SDL นี่คือ contract ของคุณ เขียนแบบ schema-first (บทเรียน [Schema-First vs Code-First](/graphql/th/tooling-federation-building/schema-first-vs-code-first/)) มันมี `Query` สำหรับการอ่านและ `Mutation` สำหรับการเขียน
- **resolvers** — ฟังก์ชันที่ทำให้แต่ละ field เป็นจริง คุณได้พบพวกมันใน [Queries & Resolvers](/graphql/th/queries-and-resolvers/) และต่อยอดด้วยการเขียนในโมดูล mutations
- **context** — object ต่อหนึ่ง request ที่ Yoga สร้างครั้งเดียวแล้วร้อยเข้าไปในทุก resolver เป็น argument ตัวที่สาม มันคือที่อยู่ของสิ่งที่มีขอบเขตต่อ request: ผู้ใช้ปัจจุบัน, handle ของ database, request id มันคือที่สะอาด ๆ สำหรับวางสิ่งที่ [Security & Performance](/graphql/th/security-and-performance/) ใส่ใจ

Yoga นำสามชิ้นนั้นมาแปลงเป็น HTTP server ที่มี GraphiQL playground ในตัว, การจัดการ error ที่สมเหตุสมผล และรองรับ subscriptions — โดยคุณไม่ต้องเขียน boilerplate เองเลย

## context ร้อยทะลุไปอย่างไร

context คือชิ้นส่วนที่มือใหม่ใช้ไม่เต็มที่ คุณนิยามฟังก์ชัน `createContext` ที่รันต่อหนึ่ง request — อ่าน header, ค้นหาว่าใครเป็นผู้เรียก — แล้วคืน object ออกมา Yoga ส่ง object นั้นให้ทุก resolver เป็น argument ตัวที่สาม ดังนั้น resolver ใด ๆ ก็อ่านได้ว่าใครเป็นคนถามโดยไม่ต้อง parse request ซ้ำ:

```ts
function createContext() {
  // Runs once per request; the return value is the resolver context.
  return { greetingPrefix: 'Hello' };
}

const resolvers = {
  Query: {
    // The third argument is the context object above.
    hello: (_parent: unknown, args: { name: string }, ctx: { greetingPrefix: string }) =>
      `${ctx.greetingPrefix}, ${args.name}!`,
  },
};
```

นั่นทำให้ resolvers สะอาดจากงานท่อต่อ request: พวกมันถาม `ctx` เอาสิ่งที่ต้องการ และการ wire อยู่ที่เดียว

## Yoga server ที่รันได้จริง — query, mutation และ context

โปรเจกต์ด้านล่างคือ `schema.ts` ที่สมบูรณ์: มัน export `typeDefs` (schema เล็ก ๆ ที่มีหนึ่ง query และหนึ่ง mutation) และ `resolvers` (พร้อม store ในหน่วยความจำและคำทักทายที่รับรู้ context) กด **Open in StackBlitz** เพื่อเปิด GraphQL Yoga server จริงขึ้นในเบราว์เซอร์ของคุณ จากนั้นเปิด GraphiQL playground ที่มันพิมพ์ออกมาแล้วลอง operation ต่าง ๆ

export const schemaCode = `// schema.ts — exports typeDefs (SDL) + resolvers for a GraphQL Yoga server.
// The fixed index.ts wraps these with createSchema + createYoga over node http.

export const typeDefs = /* GraphQL */ \`
  type Message {
    id: ID!
    text: String!
  }

  type Query {
    # A read that uses request context (see createContext below).
    greeting(name: String!): String!
    messages: [Message!]!
  }

  type Mutation {
    # A write that mutates the in-memory store and returns the new node.
    postMessage(text: String!): Message!
  }
\`;

// In-memory store — stands in for a database in this demo.
let nextId = 1;
const messages: Array<{ id: string; text: string }> = [
  { id: String(nextId++), text: 'Welcome to the graph' },
];

// Context is built once per request and threaded into every resolver as
// the third argument. Here it carries a greeting prefix; in a real server
// it would carry the authenticated user, a db handle, a request id, etc.
export function createContext() {
  return { greetingPrefix: 'Hello' };
}

export const resolvers = {
  Query: {
    greeting: (
      _parent: unknown,
      args: { name: string },
      ctx: { greetingPrefix: string },
    ) => \`\${ctx.greetingPrefix}, \${args.name}!\`,
    messages: () => messages,
  },
  Mutation: {
    postMessage: (_parent: unknown, args: { text: string }) => {
      const message = { id: String(nextId++), text: args.text };
      messages.push(message);
      return message;
    },
  },
};`;

เมื่อมันรันอยู่แล้ว ลองการอ่านก่อน:

```graphql
{ greeting(name: "Graph") messages { id text } }
```

`greeting` พิสูจน์ว่า context ไหลเข้ามา — prefix `"Hello"` มาจาก `createContext` ไม่ใช่จาก resolver จากนั้นรันการเขียนแล้วอ่านอีกครั้ง:

```graphql
mutation { postMessage(text: "Built it end to end") { id text } }
```

Query `messages` อีกครั้งแล้ว message ใหม่ของคุณก็จะอยู่ตรงนั้น ในไม่กี่บรรทัด คุณได้ออกกำลังให้ query หนึ่งอัน, mutation หนึ่งอัน, request context และ store ในหน่วยความจำ — รูปร่างแบบเดียวกับที่ Yoga service บน production ทุกตัวมี เพียงแต่เล็กกว่า

## สิ่งนี้อยู่ตรงไหนในคอร์สทั้งหมด

เซอร์วิสนี้คือจักรวาลย่อของหนังสือทั้งเล่ม `typeDefs` คือการออกแบบ schema resolvers คืองาน queries-and-resolvers และ mutations `context` คือที่ที่ security และ performance เข้ามาเกี่ยวข้อง เพิ่ม codegen และ test จากช่วงก่อนหน้าในโมดูลนี้ แล้วคุณก็ได้เซอร์วิสที่มี type ผ่านการ test แยกมันตาม domain แล้ววาง gateway ไว้ข้างหน้า แล้วคุณก็ได้บทเรียน federation ไฟล์เล็ก ๆ ไฟล์เดียว ทุกแนวคิด

<Callout title="คุณสร้าง GraphQL service ได้แบบ end to end">
GraphQL service เป็นเพียงสามสิ่งที่ผูกเข้าด้วยกัน: **typeDefs** สำหรับ contract, **resolvers** เพื่อทำให้มันเป็นจริง และ **context** เพื่อร้อย state ต่อ request GraphQL Yoga เปลี่ยนสิ่งเหล่านั้นให้เป็น HTTP server จริงที่มี playground และไม่มี boilerplate ทุกอย่างที่เหลือในคอร์สนี้ — design, codegen, testing, federation, security — แขวนอยู่บนโครงนี้ คุณเริ่มจากศูนย์ด้วย type เดียว ตอนนี้คุณตั้ง GraphQL service ที่มีเอกสาร มี type ทดสอบได้ และประกอบได้ขึ้นมาได้แล้ว นั่นแหละคือส่วน hero
</Callout>

export const yogaFlow = `flowchart TD
  TypeDefs["typeDefs (SDL contract)"]
  Resolvers["resolvers (fulfil each field)"]
  Schema["Executable schema (createSchema)"]
  Context["createContext (per request)"]
  Yoga["createYoga"]
  Http["HTTP server + GraphiQL"]
  TypeDefs --> Schema
  Resolvers --> Schema
  Schema --> Yoga
  Context --> Yoga
  Yoga --> Http`;
